# Solution
1. Просто распарсил файл (точнее, *прочитал построчно*), посчитал числа, которые спрашиваются в условии, ничего биологичного в этом
пункте.

2. GC-content посичтал через подсчет количества символов G и C в прочтениях, для этого воспользовался
std::count_if из <algorithm>, чтобы не писать цикл.

3. Phred-quality на 10 позиции посчитал, взяв 10 позицию в каждой строке качества, если её длина позволяла.

4. Этот пункт был самым сложным, я довольно долго получал WA, потому что не понимал нескольких вещей:
- прочтение называется **подверженным триммингу, если первое же окно в его строке качества оказывается хуже, чем ожидаемое качество**, либо же если длина строки качества меньше, чем ширина окна.
- при тримминге окном прочтение считается не валидным, **если его среднее качество <= baseline**, хотя по-моему логично было бы просто <.
- второй тримминг проводился по уже отфильтрованному первым триммингом файлу, **именно по неравенству на длину: string.size() < 60** (такие строки считаются невалидными)
В остальном в этом пункте ориентировался на алгоритм из trimmomatic и то, что рассказывали на паре.

# Features
- find_trimmed_index(const std::string_view &,
                                     uint16_t, double) принимает string_view для того, чтобы можно было считать статистику для файлов, полученных в результате нескольких разных триммингов, не копируя при этом ни строки. Также метод принимает именно double (и uint16_t) для возможностей расширения/
